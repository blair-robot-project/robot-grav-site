### 2026-07-11 — 🚀 LIVE: automated the "2026 Robojawn" banner to update itself from live TBA match results
Built a live-updating banner for RoboJAWN 2026 (the team's July 10-12 Philadelphia regional): the existing static "2026 Robojawn" `banners-data` entry (green, generic "follow along" text, linking to the event's thebluealliance.com page) now gets rewritten automatically — yellow, with the actual won/lost result and record — every time a new frc449 match result posts to TBA, via a standalone script polling TBA's `/team/frc449/event/2026paphi1/status` endpoint every 15 minutes through the event.
- **Runs off-repo, on a separate always-on machine, not this site's server.** The automation lives on Brad's personal M1 MacBook Air (`~/robojawn-449-banner/banner_watch.py` + a bounded `launchd` schedule — 15-min ticks, 8am–8pm ET, July 11–12 only, self-expiring) — documented in Brad's own `bpeniston/dev-infrastructure` repo, not here, since it's infrastructure unrelated to this site's hosting.
- **Talks to the live site over the same REST API the `grav-live` Claude MCP connector uses** (`https://robot.mbhs.edu/api/v1`, `X-API-Key` auth, `GET`/`PATCH /pages/{route}`) but calls it directly with Python's stdlib `urllib` — no Claude Code or MCP needed on that machine at all. Endpoint shape (base URL, auth header, `If-Match`/ETag semantics, and that `header` fields — including list fields like `banners` — get replaced wholesale on `PATCH`, not deep-merged element-by-element) came from reading the `grav-mcp` connector's own source (`~/Documents/449/grav-mcp/src/client/grav-client.ts`) rather than guessing. See RUNBOOK.md's Cautions & Gotchas for the reusable pattern.
- **Gotcha worth keeping:** TBA's Cloudflare WAF blocks Python's default `urllib` User-Agent outright (403 "Access denied... browser's signature") even with a valid API key — a `curl`-like `User-Agent` header gets through fine.
- **Tried and discarded two other mechanisms first:** a Claude Code desktop "Routine" (worked, but only fires while that app/laptop stays open and awake) and a claude.ai cloud `RemoteTrigger` (dead end — that environment's available MCP connectors don't include `grav-live`/`grav-staging` at all, since those are configured per-machine locally, not account-wide).
- **End state:** banner flips to yellow with the match-by-match record on the first real result; `end_date` set to 2026-07-14 so it removes itself; a final overall-performance summary posts once the event ends, after which the `launchd` schedule (bounded to exactly the event window) goes dormant with nothing further to clean up.
- **Added same day: a text to Brad on every update**, via AppleScript/iMessage from the Air's own Messages account — same message text as the banner. Needs an unlocked, actively logged-in GUI session on the Air to work at all (first attempt hung with an `AppleEvent timed out` while the screen was locked); if it locks again mid-event, texts silently stop while the banner keeps updating fine regardless. See RUNBOOK.md's Cautions & Gotchas.

### 2026-06-25 — 🧪 SUBDOMAIN (449.navybook.com — admin2 `/sync` → 403 shim PROTOTYPED in `.htaccess`)
Prototyped option 1 of the admin2-collab-path decision: force the unimplemented `/api/v1/sync/*` endpoints to return **403** (not 404) so admin2 takes its clean solo-mode fallback. Added a rule block to the trial's `.htaccess` (backup `.htaccess.bak-20260624-*`).
- **DreamHost `.htaccess` gotcha discovered (cost most of the session).** On this host, `RewriteCond %{REQUEST_URI} …` conditions **and start-anchored per-directory `RewriteRule` patterns (`^api/v1/sync…`, `^zzzforbidden$`) silently never match** — requests fall through to Grav. The shipped Grav rules work only because they match on `%{QUERY_STRING}` or use the catch-all `.*`. Proven by isolated probes: a `[F]` rule keyed on `%{REQUEST_URI}` → 404, the identical rule keyed on **`%{THE_REQUEST}`** (the raw request line) → 403. So the working rule matches the request line:
  ```apache
  RewriteCond %{THE_REQUEST} \s/+api/v1/sync(/|\s|\?) [NC]
  RewriteRule .* - [F,L]
  ```
- **Verified (HTTP level):** `/api/v1/sync`, `/api/v1/sync/capabilities`, `/api/v1/sync/channels/pull`, `/api/v1/sync/pages/home/pull`, `/api/v1/sync/?x=1` → all **403**. Boundary intact: `/api/v1/sync-not-really` and `/api/v1/info` → 401 (unaffected), site + `/admin` → 200, YouTube iframes still render. Probe lines + test marker removed; file clean.
- **✅ Validated in-browser 2026-06-25 (Chrome ext., logged-in admin).** On a clean editor boot admin2 fires a **one-time** `/sync` probe burst — `GET /sync/capabilities`, `GET /sync/channels/pull`, `POST /sync/pages/community/fll-team/{pull,init,presence}` — **all 5 return 403**, then it **stops** (no flood). ### 2026-06-24 — 🧪 SUBDOMAIN (449.navybook.com — YouTube iframes FIXED: Grav 2.0 GFM tagfilter)
